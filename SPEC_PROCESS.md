## 一、brainstorming 关键节点

### 智能体追问的好问题

| 轮次 | 智能体的问题 | 我的回答 | 如何影响了设计 |
|------|-------------|----------|---------------|
| 1 | LLM 调用成本怎么处理？——A 用户自带 Key / B 后端统一免费+付费 / C 完全免费自部署 / D 混合 | "哪个简单" → A | 最初选了 A（用户自带 Key），后来改为后端统一调用，简化了用户门槛 |
| 2 | 用什么技术栈？——A FastAPI / B Node.js / C Next.js | 智能体推荐 Next.js，我改为 Vue + Node | 决定前端 Vue + 后端 Node.js（Express），前后端分离 |
| 3 | 公开梦境接力续写模式？——A 线性接力 / B 分支叙事 / C 独立续写 | 线性接力 + 独立续写都要 | 增加产品复杂度：continations 表需同时支持 parent_id 链条和 is_independent 标记 |
| 4 | LLM 续写触发方式？——A 一键续写 / B 逐段交互 / C 后台静默 | A 一键续写 | 简化交互，LLM 一次生成完整故事 |
| 5 | 用户体系？——A 完整注册登录 / B 匿名 / C 轻度登录 | A 完整注册登录 | 增加 Auth 模块 |
| 6 | 碎片串联模式？——A 全自动 / B 用户指导 / C 两者都有 | C 两者都有 | fragment 表需要 sort_order 字段支持手动排序 |
| 7 | 分享卡片形式？——A 后端生成图片 / B HTML 卡片 / C 前端 Canvas | "哪个简单" → B | 前端 HTML 渲染卡片，后来加上了 html2canvas 一键导出 |
| 8 | LLM 服务商？——A OpenAI / B Claude / C 两者 / D OpenAI 兼容通用 | D | 用 OpenAI 兼容接口，覆盖最多后端 |
| 9 | 数据库？——A PostgreSQL / B MongoDB / C SQLite / D Prisma+PG | "不想配置" → C SQLite | 零配置，影响 ORM 选型（不能用 Prisma） |

### 我修正原有设想的关键时刻

| 轮次 | 原设想 | 修正后 | 触发原因 |
|------|--------|--------|----------|
| 1 | 没有明确技术栈 | 确定 Vue.js + Node.js | 智能体推荐 Next.js 后我根据自己的偏好修改 |

## 二、关键迭代记录（至少 3 轮）

### 第 1 轮迭代：技术选型与核心决策

- **对话节选**：
  ```
  智能体：LLM 成本怎么处理？
  我：哪个简单
  智能体：A 最简单（用户自带 Key）
  我：前面内个 AI 改成后端 API 吧，我直接提供，目前不做额度
  智能体：好，LLM 调用改由后端统一处理
  ```
- **我的决策**：最初接受"用户自带 Key"，后来想了一下改为"后端提供 Key"。因为课程项目阶段不想让用户配置 API Key 这个门槛。
- **结果**：架构简化——不需要在前端管理用户的 API Key 存储和安全传输。后端环境变量设好就行。

### 第 2 轮迭代：技术栈与数据库

- **对话节选**：
  ```
  智能体：推荐 Next.js，前后端一体，Vercel 免费部署
  我：前端 vue 后端 node.js 吧
  智能体：（接受，调整为 Vue + Node）
  
  智能体：数据库？A PostgreSQL / B MongoDB / C SQLite / D Prisma+PG
  我：没有并发，我就是做，我不想配置
  智能体：SQLite，不需要安装数据库服务器，一个文件就能跑
  ```
- **我的决策**：坚持用自己熟悉的 Vue + Node 而不是智能体推荐的 Next.js；数据库选最省事的 SQLite。
- **结果**：前后端分离的项目结构，SQLite 单文件数据库。这在课程项目场景下完全合理。

### 第 3 轮迭代：产品功能细节

- **对话节选**：
  ```
  智能体：公开梦境接力续写——A 线性接力 / B 分支叙事 / C 独立续写？
  我：有一个线性接力的，然后还有一个独立续写的
  智能体：两种模式共存
  ```
- **我的决策**：在智能体给的三个选项中选了 A+C 组合，增加了功能复杂度但提升了产品体验。
- **结果**：continuations 表设计需要同时处理两种模式——parent_id 做链条，is_independent 做标记。

## 四、writing-plans 过程记录

### 计划拆解思路

将 SPEC 中 6 个功能模块拆解为 30 个 task，按 TDD 红-绿-重构节奏组织：

| 阶段 | Task | 内容 |
|------|------|------|
| 阶段 1 | T01-T03 | 项目脚手架（后端/前端/数据库）——全并行 |
| 阶段 2 | T04-T08, T16 | Auth 全链路 + 标签 + LLM 服务 ——多分支并行 |
| 阶段 3 | T09-T18 | 梦境 CRUD + AI 续写 |
| 阶段 4 | T19-T26 | 接力续写、碎片串联、收藏 ——三个功能分支可并行 |
| 阶段 5 | T27-T30 | 分享卡片、导航栏、Docker、CI ——收尾 |

### plan 的特点

- 每个 task 5 步（写测试 → 跑红 → 实现 → 跑绿 → commit），颗粒度 2-5 分钟
- 所有 step 包含完整代码，无 TBD/TODO
- 前端/后端任务交叉出现，按依赖链排列
- 阶段 1-2 可大量并行，最大化利用 worktree 隔离

### 对 SPEC 的反向修正

写 plan 过程中发现并修正了一个 SPEC 遗漏：
- SPEC 中 `fragments` 表结构需要 `session_id` 字段来分组，已在 T22 代码中体现
- 前端路由新增 `/fragments/:id` 用于查看历史串联详情，SPEC 未明确列出，已补充

### 解读不一致的地方

| 我的原意 | agent 的解读 | 是 spec 写错了还是它读错了？ |
|----------|------------|--------------------------|
| `visibility` 字段的默认值：录入时如果不传，默认 `private` | agent 理解为需要前端显式传 `visibility: 'private'`，否则拒绝请求 | SPEC §6 数据模型写了 `DEFAULT 'private'`，但 API 设计章节没有说明这个字段是可选的。agent 从 API 角度理解"没写 optional = required"是合理的。**SPEC 写得不完整。** |
| `GET /api/v1/dreams` 的认证行为：我本意是未登录只能看公开，登录后能看到自己的私有 + 所有公开 | agent 理解成"只有传了 ?visibility=public 才返回公开，否则返回空" | SPEC §3 模块 B 和 API 设计都没描述 GET /dreams 的权限逻辑。**SPEC 遗漏了权限矩阵。** |
| 梦境续写的结果存储：我预期 `ai_story` 是 TEXT 字段存整个故事 | agent 以为需要分段落存储或者在 `continuations` 表里再存一份 | SPEC 数据模型清晰标注了 `ai_story TEXT`，但 agent 看到 `continuations` 表也有 `content`，混淆了两个概念。**Agent 读错了**——但提示 SPEC 应该加一段区分 "AI 续写（ai_story，私有）" 和 "用户接力（continuations，公开）"。 |

### 产出代码与预期的差距

| 维度 | 预期 | 实际产出 | 差距原因分析 |
|------|------|---------|-------------|
| 功能完整性 | T09 应包含 POST /dreams + 标签自动创建 | Agent 实现了 POST /dreams 但跳过了自定义标签自动创建逻辑 | PLAN T09 Step 3 代码中有标签处理，但 agent 说"这部分代码看起来像是可选功能"——因为 SPEC 没有把它列为 P0 需求 |

| 测试质量 | 3 个测试：成功创建、401 未认证、400 空内容 | Agent 只写了 1 个测试（成功创建） | PLAN 明确列出了 3 个测试用例，但 agent 说"先写一个验证流程跑通"。**Agent 没有严格遵循 TDD 要求**——这暴露了 PLAN 中没有显式写"必须写 3 个测试，缺一不可"的约束 |

## 七、迭代补记（初版之后）

初版 SPEC 在后续迭代中扩展了 7 类功能（详见 `SPEC.md §11`）。回看 brainstorming 阶段，可以补上当初留空的两条：

**让我不满的地方（补）**：brainstorming 只锁定了 MVP 边界，没有预判"社交 / 通知 / 数据分析"这类几乎必然出现的延伸方向。结果迭代时加 `follows`、`notifications`、`likes`、`comments` 都要新开表和路由，初版数据模型完全没有为关系型/事件型数据留白。

**改进建议（补）**：SPEC 阶段可加一节"可预见的扩展方向"——即使不实现，也让初版的数据模型与接口为之留余地（如预留用户关系、事件流抽象），显著降低后续迭代的结构性改动成本。

**一个正向印证**：brainstorming 当初坚持的"最简方案优先"（SQLite、后端统一 Key、纯 Vue 组件）在迭代中没有成为障碍——新增 4 张表、10 个端点都平滑落地，说明"先简后扩"的取舍在本项目是成立的。
