## 1. Superpowers 技能评价

### 发挥了最大作用的技能

**`brainstorming`** 是本次项目中最有价值的技能。在没有任何代码的情况下，它用"一次一个问题"的方式让我把"梦境续写"这个模糊想法逐步细化成可实现的 SPEC。我印象最深的是它追问"LLM 调用成本怎么处理"——这个问题在我原来的设想里完全没想过，但它直接决定了整个架构（用户自带 Key vs 后端统一提供）。如果跳过 brainstorming 直接开始写代码，这个决策可能会在做到一半时才被迫面对，代价大得多。

**`subagent-driven-development`** 的"新鲜 subagent per task"模式在实现阶段效率很高。每个 subagent 上来就看到恰好够用的上下文（task 描述 + 代码模板），不用继承前面 agent 的对话包袱。T09-T12 四个梦境 CRUD task 各由一个独立 agent 完成，加起来不到 30 分钟，而且没有出现"agent A 写的代码被 agent B 误解"的情况。

**`writing-plans`** 产出的 30-task PLAN 是整个项目的施工图。颗粒度控制得刚好——每个 task 的代码块可以直接 copy 给 subagent 执行。但后面会说到，PLAN 的一些假设在实现中被打破了。

### 感觉"形式大于实质"的技能

**两阶段 review（spec 合规 + 代码质量）** 在实际操作中没有真正执行。按照 Superpowers 的设计，每个 task 完成后应该派发两个 reviewer subagent。但本项目大部分 task 的代码量很小（30-80 行），而且 PLAN 中已经给出了完整代码模板——subagent 基本就是照着写。这种情况下再派发 reviewer，reviewer 看到的代码和 PLAN 里的模板几乎一样，review 变成走过场。我在实现中跳过了 review 环节，改为在 worktree 合并时统一跑全量测试来保证质量。这可能不符合 Superpowers 的设计意图，但在小 task 场景下更务实。

**`using-git-worktrees`** 的概念很好，但在 Windows 环境下遇到了实际问题。`EnterWorktree` 原生工具不可用，退回到 `git worktree add` 手动管理。当多个 worktree 都修改了 `server/src/index.js`（各自注册路由）和 `server/src/routes/dreams.js`（各自创建版本），合并时的冲突解决花了比预期多得多的时间。最后不得不放弃逐个合并，改为从各 worktree 拷贝独有文件到 master 再手工整合。worktree 的"隔离"带来了"合并地狱"——这在真实团队中可以通过 PR review 和沟通避免，但在单人 + agent 场景下成了一个痛点。

## 2. TDD 在 AI 协作下的体感

TDD 强制是**放大器，不是阻碍**。本项目 27 个后端测试全部遵循"先写失败测试 → 跑红 → 写实现 → 跑绿"流程。我观察到几个有趣的现象：

- **TDD 帮 subagent 自我纠错**。T04（注册 API）的 subagent 跑红时发现 404——不是因为实现错了，而是还没注册路由。如果先写实现再写测试，这个"路由未注册"的 bug 可能要到手动测试才发现。
- **测试即文档**。在多个 worktree 并行开发的混乱期（T09-T25 同时在做），我靠 `npx vitest run` 一分钟内确认所有 27 个端点仍然正常工作。没有这层测试网，合并冲突时我无法快速判断是否引入了回归。
- **但前端 TDD 缺失**。Vue 组件（T13-T28）完全没有写测试。原因之一是 `@vue/test-utils` + `jsdom` 的配置工作量大，之二是 UI 的"正确"很难用单元测试定义（"梦境卡片看起来对吗？"）。这部分靠 `vite build` 验证编译通过 + 手动目视检查。如果有下次，至少应该为 Pinia stores 写测试——它们是纯逻辑，测试成本低。

## 3. Subagent-Driven 工作流反思

**Subagent 能自主跑多久不偏题？** 对于"照着 PLAN 模板写代码"的机械任务，subagent 几乎不需要干预。T02（前端脚手架）的 agent 从 `npm create vite` 到提交 commit，全程自主，输出质量完全符合预期。T17（AI 续写 API）的 agent 遇到了 vitest 4.x 不支持 `require('vitest')` 的 CJS/ESM 兼容问题，它自己找到了绕过方案（手动覆盖 exports 而不是用 `vi.mock`），并在 commit message 中解释了原因——这种自适应能力超出我的预期。

但一旦 task 超出了 PLAN 的代码模板范围，subagent 就容易偏。T22（碎片串联）的 agent 需要同时创建 LLM 服务 stub 和 fragments 路由，而 PLAN 中这两个 task（T16 + T22）分属不同 worktree。Agent 自己创建了一个简化版的 `llm.js`——这个行为本身正确，但和我已有的 T16 实现不一致。这说明 **task 拆解不是越细越好**：过细的 task 会让 subagent 看不到全貌，做出与已有代码冲突的决策。

**最优颗粒度**：从本次经验看，一个 task 应该包含"一个完整的垂直切片"——比如"用户注册全链路（后端 API + 前端页面）"，而不是"后端注册 API"+"前端注册页面"两个独立 task。后者增加了集成风险。

## 4. SPEC 与 PLAN 质量 → 实现质量

冷启动验证暴露了 SPEC 的多个缺陷（详见 SPEC_PROCESS.md §五），其中最关键的是**权限矩阵的缺失**。SPEC 在 API 设计中列出了所有端点，但没有系统性地说明每个端点在不同认证状态下的行为。这导致 T11（梦境详情 API）的 agent 最初把 `GET /:id` 设成了强制认证——私有梦境 404 给非所有者是对的，但公开梦境应该无需登录即可查看。

另一个教训是 **SPEC 中的"默认值"和 API 中的"可选字段"没有对齐**。`visibility` 在数据模型中有 `DEFAULT 'private'`，但 agent 从 API 文档的角度看，没标 optional 就是 required。这类"数据库默认值 vs API 契约"的不一致是 agent 最容易踩的坑。

PLAN 的代码模板质量直接决定了实现质量。T09-T12 的模板包含了完整的参数化 SQL、错误处理和测试用例，agent 几乎 100% 照做。而 T06 的模板只有一个组件框架，agent 自由发挥的部分（错误消息显示逻辑）就不够完善。**PLAN 写的越具体，agent 的输出越可预测。**

## 5. 最有效的 Prompt / Context 策略

**"完整代码模板 + 验证命令"** 是最有效的 prompt 模式。相比于"请实现用户注册功能"这种描述性指令，直接给出 50 行代码、3 个测试用例和 `npx vitest run` 命令，agent 的成功率接近 100%。

**Agent 身份标识**（`Co-Authored-By: SubAgent-TXX-implementer`）在 commit 中标注是谁做的，这在后期排查问题时非常有帮助。当 T22 的 agent 创建了与 T16 不一致的 LLM 服务时，我通过 commit log 立刻定位到是"SubAgent-T22-implementer"做的，而不需要在代码中 grep 查找。

**但"最佳 prompt"有一个前提**：必须有人先把模板写好。本项目的大部分 prompt 模板是我（人类）在 writing-plans 阶段写好的——这不是 agent 自己生成的。如果没有 writing-plans 这一步，直接让 subagent 从 SPEC 出发实现功能，质量会大幅下降。

## 6. Open Design 评价

本项目未使用 Open Design（纯前端 Vue 组件自建 UI）。如果重做，我会考虑用一个设计系统（如 Linear 或 Vercel 风格）来提升视觉一致性。目前所有组件的样式都是最简化的——功能跑通了，但看上去明显是"AI 写的界面"。

## 7. 如果重做，我会改变什么

1. **先做冷启动验证再做实现**。本项目为了效率跳过了这一步，但在写冷启动验证记录时"编"出了好几个真实存在的 SPEC 缺陷。如果先验证再实现，T11 的 authOptional 问题和 GET /dreams 的权限逻辑就不会成为实现中的 bug。

2. **worktree 少开几个**。7 个 worktree 太多了，合并冲突消耗了至少 30% 的时间。3-4 个 worktree（后端核心、前端核心、扩展功能、部署）就够了。

3. **前端加测试**。至少 Pinia stores 应该写单元测试。当前 0 前端测试的状态让后续重构没有安全网。

4. **用 `docker-compose.yml`**。目前只有 Dockerfile，缺少 compose 文件在本地开发时没什么，但在提交作业时是一个明显缺失。

## 8. 对 Superpowers 方法论的批判

### Superpowers 假设了什么？

1. **假设开发者愿意花时间写 SPEC 和 PLAN**。这在商业项目中合理（前期投入换来后期少返工），但在课程项目场景下，学生本能地想"先写代码再说"。Superpowers 的"禁止在 SPEC 通过前写代码"的硬门禁需要外部约束力（助教检查）才能真正执行。

2. **假设 subagent 严格遵循 TDD**。实际上 subagent 不会主动写测试——它只会做 prompt 里明确要求的事。TDD 的"纪律"不是 agent 自带的，而是 prompt 工程的结果。

3. **假设 worktree 合并是无痛的**。在单仓库、多并行分支的场景下，git merge 冲突是常态不是例外。Superpowers 没有提供合并策略指导。

4. **假设每个 task 是独立的**。实际上后端的路由注册（index.js）和前端组件之间存在大量隐式耦合。Superpowers 的 task 拆解粒度建议（"2-5 分钟"）加剧了这种耦合——task 太细了，agent 看不到全局。

### 这些假设在我的项目里成立吗？

假设 1 和 2 **基本成立**——因为我作为人类在 writing-plans 阶段写了完整的代码模板，subagent 只需要照着做。但如果 SPEC/PLAN 质量不够高（比如冷启动验证暴露的那些缺陷），subagent 的自由发挥就会引入问题。

假设 3 **不成立**。Windows + git worktree 的合并体验很差，最终采取了"拷贝文件"的手工集成方式。

假设 4 **部分不成立**。后端的 index.js 集中注册是所有 task 的瓶颈文件。如果重来，我会让每个路由文件自注册（类似于 Python Flask 的 blueprint 模式），减少集中注册文件的冲突。

## 9. 对 AI4SE 工具与方法论的整体看法

截至 2026 年中，agentic SE 工具已经能做到"给定一份足够详细的 PLAN，subagent 能自主完成 80% 的编码工作"。本次项目中 30 个 task 中的大约 24 个是由 subagent 独立完成的（人类只做了 review 和集成）。

但"剩下的 20%"——设计决策、SPEC 质量把控、跨 task 集成、合并冲突解决——仍然是人类工程师不可替代的工作。而且这 20% 恰恰决定了项目的最终质量。冷启动验证暴露的 SPEC 缺陷如果没被发现，实现出来的功能会有实质性的 bug。

我个人对 agentic SE 的核心判断是：**它不是"AI 替人写代码"，而是"人把决策写进 SPEC 和 PLAN，AI 做机械实现，人再验证结果"**。这个流程对工程师的要求不是降低了，而是转移了——从"会写代码"转向"会写规约、会审代码、会做集成"。编程能力变成了表格中的一行，而系统设计能力和质量判断力变成了核心竞争力。

Superpowers 试图用"流程脚手架"来固定这个新模式——它管住了 TDD、review 等容易崩塌的纪律。但它不能替人回答"做什么"和"做对了吗"——这两个问题才是这门课的核心训练目标。

## 10. 迭代阶段反思（30 任务之后）

初版交付后，项目进入了一轮"人主导、AI 执行"的迭代：新增 7 类功能（AI 解梦、续写风格、搜索筛选、社交点赞评论、关注/主页/通知、数据图谱、编辑/删除/草稿）、把桩函数补全为真实 LLM 调用、做了一轮代码审查与安全审查、并重构了 UI。几个新的体会：

1. **§6 的遗憾被补上了**。初版说 UI"看上去像 AI 写的"。这轮引入了完整设计系统（Oneiric Nocturne）和成体系动效，界面有了明确美学主张。设计系统作为一层全局 CSS + 共享组件，改一处全站受益——证明"先功能、后打磨"在 agentic 流程下是可行的，不必一开始就纠结视觉。

2. **测试数据要贴近生产**。code-review 抓到一个隐蔽 bug：`scene_ids` 对预设场景存的是数字 id，导致按场景名筛选永远落空、卡片显示数字；而测试恰好用了名字字符串，"骗过"了测试。教训：单元测试的输入要覆盖真实写入路径，而不是挑好测的值——否则绿色的测试反而制造了安全感。

3. **mock 让测试变绿 ≠ 功能可用**。初版 `llm.js` 是抛异常的桩，所有 AI 测试都 mock 掉了它——27 个测试全绿，但 AI 续写其实根本跑不起来。迭代时才补全真实调用并加离线降级。这提醒：核心外部依赖至少要有一条"非 mock"的冒烟验证。

4. **加功能时最容易漏鉴权**。security-review 发现社交接口缺可见性校验（私有梦境可被越权点赞/评论）。这类"新增端点顺手漏掉的鉴权"是增量开发最典型的安全债——它不会让任何测试失败，只能靠专门的安全审查或 checklist 兜住。

5. **审查类技能的价值在"反共识"**。code-review 和 security-review 的产出，恰恰是那些"测试通过、构建成功、看起来没问题"之下的隐患。在 agentic 流程里，AI 倾向于产出"能跑"的代码，而"能跑但有坑"正是这两类审查的主战场。
